# Local Development Setup

This guide covers the complete steps to run the GP2F monorepo on a development machine. Follow the sections in order. Each section assumes the previous sections have been completed successfully.

---

## Prerequisites

The following tools must be installed before proceeding. Versions listed are the minimum tested versions.

**Rust toolchain:** 1.78.0 or later, with the `wasm32-unknown-unknown` target.
**wasm-pack:** 0.12.1 or later.
**Node.js:** 20.0.0 or later (LTS recommended).
**pnpm:** 9.0.0 or later.
**Docker and Docker Compose:** Docker Engine 24.0 or later, Compose plugin 2.24 or later.
**protoc:** 3.21 or later (Protocol Buffers compiler).
**Temporal CLI:** 0.13.0 or later.

On macOS, all of the above can be installed via Homebrew. On Linux, use the system package manager for Docker and download the others from their official release pages.

---

## Step 1: Clone and Initialize the Repository

```bash
git clone https://github.com/your-org/gp2f.git
cd gp2f
```

Install all JavaScript dependencies from the monorepo root. GP2F uses `pnpm` workspaces.

```bash
pnpm install
```

Verify that the Rust toolchain is present and the WASM target is available.

```bash
rustup show
rustup target list --installed | grep wasm32
```

If `wasm32-unknown-unknown` is not listed, add it:

```bash
rustup target add wasm32-unknown-unknown
```

---

## Step 2: Build the Policy Core WASM Package

The `policy-core` crate must be compiled to WASM before the React frontend can start. This compilation is the most time-intensive step in the setup process and takes approximately 90 seconds on a modern laptop.

```bash
cd policy-core
wasm-pack build --target bundler --out-dir ../client-sdk/wasm-out
cd ..
```

After this command completes, the directory `client-sdk/wasm-out/` will contain:

- `policy_core_bg.wasm` — the compiled WASM binary
- `policy_core.js` — the JavaScript bindings generated by `wasm-pack`
- `policy_core_bg.wasm.d.ts` and `policy_core.d.ts` — TypeScript type definitions

Do not modify files in `wasm-out/`; they are regenerated on every WASM build.

---

## Step 3: Start Infrastructure Services

GP2F requires PostgreSQL, Redis, and the Temporal server for local development. All three are provided as Docker Compose services.

```bash
docker compose up -d postgres redis temporal
```

Wait for all three services to report healthy status:

```bash
docker compose ps
```

The expected output shows `postgres`, `redis`, and `temporal` all with status `healthy`. If any service shows `starting`, wait 10–15 seconds and run the command again.

**PostgreSQL** is exposed on `localhost:5432` with credentials `gp2f_dev / gp2f_dev_secret` and a database named `gp2f_dev`.

**Redis** is exposed on `localhost:6379` with no authentication (development only; authentication is required in production).

**Temporal** is exposed on `localhost:7233` (gRPC) and `localhost:8088` (Web UI). Navigate to `http://localhost:8088` in a browser to verify the Temporal server is running. You should see the Temporal Web UI with no workflows in the default namespace.

---

## Step 4: Run Database Migrations

Apply the PostgreSQL schema migrations using the `sqlx` CLI tool.

```bash
# Install sqlx-cli if not already installed
cargo install sqlx-cli --features postgres

# Run migrations
DATABASE_URL=postgres://gp2f_dev:gp2f_dev_secret@localhost:5432/gp2f_dev \
  sqlx migrate run --source server/migrations
```

After the migrations complete, verify the schema by connecting to the database:

```bash
psql postgres://gp2f_dev:gp2f_dev_secret@localhost:5432/gp2f_dev -c '\dt'
```

You should see the following tables: `events`, `policy_versions`, `sessions`, `yrs_snapshots`.

---

## Step 5: Seed Development Data

The seed script creates a test policy AST, two test users, and a sample document.

```bash
cargo run --bin seed -- --env development
```

On success, the script prints the IDs of the created resources. Note the `policy_version_id` and a `user_id` value — you will need them to construct test requests in later steps.

---

## Step 6: Start the Axum Server

Open a new terminal tab or pane. The server uses environment variables for configuration. For local development, these are loaded from a `.env` file.

```bash
cp server/.env.example server/.env
```

Review `server/.env` and confirm the values match your local infrastructure (they should by default). Then start the server:

```bash
cd server
cargo run --release
```

The `--release` flag is recommended for local development because the WASM evaluator (Wasmtime) performs significantly better in release mode. Debug builds will work but will be noticeably slower.

The server starts on `http://localhost:3000`. It logs startup events to stdout. Confirm you see the following lines before proceeding:

```
INFO  gp2f_server: Axum listening on 0.0.0.0:3000
INFO  gp2f_server: Wasmtime evaluator initialized, policy_version=v2.3.1
INFO  gp2f_server: Temporal client connected to localhost:7233
INFO  gp2f_server: Redis PubSub connected to localhost:6379
```

If you see an error about a missing `policy_version`, ensure Step 5 (seed) completed successfully.

---

## Step 7: Start the Temporal Worker

Open another terminal tab. The Temporal worker process runs the reconciliation workflow activities.

```bash
cd server
cargo run --bin worker
```

Confirm you see:

```
INFO  gp2f_worker: Temporal worker started, namespace=default, task_queue=gp2f-reconcile
INFO  gp2f_worker: Registered activity: reconcile_op_id
INFO  gp2f_worker: Registered activity: compact_yrs_snapshot
```

---

## Step 8: Start the React Frontend

Return to the monorepo root. The frontend development server is a Vite application.

```bash
cd client-sdk
pnpm dev
```

The Vite server starts on `http://localhost:5173`. Open this URL in a browser. You should see the GP2F development harness with a document editor, a policy inspector panel, and a real-time sync log.

---

## Step 9: Verify the Complete Stack

With all services running, perform the following smoke test to verify that the full stack is operating correctly.

1. In the browser at `http://localhost:5173`, select the test document seeded in Step 5.
2. Make a change to the document title field.
3. Observe the sync log panel: you should see an entry showing the `op_id` emitted, and within approximately 100ms, an `ACCEPT` response from the server.
4. Open the Temporal Web UI at `http://localhost:8088`. You should see a completed workflow run for the `reconcile_op_id` workflow.
5. In the psql shell, run: `SELECT outcome FROM events ORDER BY created_at DESC LIMIT 1;` You should see `ACCEPT`.

If all five checks pass, the complete development stack is running correctly.

## Environment Variables Reference

The following variables are defined in `server/.env.example`:

| Variable | Default | Description |
|---|---|---|
| `DATABASE_URL` | `postgres://gp2f_dev:gp2f_dev_secret@localhost:5432/gp2f_dev` | PostgreSQL connection string |
| `REDIS_URL` | `redis://localhost:6379` | Redis connection string |
| `TEMPORAL_HOST` | `localhost:7233` | Temporal server gRPC address |
| `WASM_POLICY_PATH` | `../client-sdk/wasm-out/policy_core_bg.wasm` | Path to the native WASM file for Wasmtime (built by `wasm-pack` in Step 2) |
| `DEVICE_SESSION_HMAC_SECRET` | `dev-secret-change-in-production` | HMAC secret for `op_id` generation |
| `LOG_LEVEL` | `info` | Tracing log level (`trace`, `debug`, `info`, `warn`, `error`) |
| `PORT` | `3000` | HTTP/WebSocket server port |

## Common Issues

**`wasm-pack build` fails with "error[E0463]: can't find crate for std"**

This indicates the `wasm32-unknown-unknown` target is not installed. Run `rustup target add wasm32-unknown-unknown` and retry.

**Docker Compose services fail to start on Apple Silicon (M1/M2/M3)**

The official Temporal Docker image may not have an ARM64 build available for all versions. Use the `--platform linux/amd64` flag if needed, or update the `docker-compose.yml` to pin a version known to have multi-arch support.

**`sqlx migrate run` fails with "no migrations found"**

Verify you are running the command from the monorepo root and that the `--source` path is correct. The migrations directory is `server/migrations`, not `migrations`.

**Server fails to start with "Wasmtime evaluator failed: policy binary not found"**

The WASM package has not been built or the path in `WASM_POLICY_PATH` is incorrect. Complete Step 2 and verify the path in `server/.env` points to `../client-sdk/wasm-out/policy_core_bg.wasm`.

**WebSocket connection refused in the browser**

Verify the Axum server is running and listening on port 3000. Check that the `VITE_WEBSOCKET_URL` variable in `client-sdk/.env.development` is set to `ws://localhost:3000/ws`.
